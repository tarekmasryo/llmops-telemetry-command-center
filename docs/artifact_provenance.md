# Artifact Provenance

## Source of truth

The dashboard consumes synthetic telemetry data and precomputed artifacts generated by the companion notebook workflow.

Companion notebook repository:

```text
https://github.com/tarekmasryo/llm-production-telemetry
```

The dashboard does not regenerate modeling artifacts at runtime. It loads them, validates the minimum contracts, and presents them as evidence.

## Data files

| File | Role |
|---|---|
| `data/llm_system_interactions.csv` | Main row-level synthetic LLM interaction telemetry. |
| `data/llm_system_sessions_summary.csv` | Session-level aggregate telemetry. |
| `data/llm_system_users_summary.csv` | User-level cohort and risk summary. |
| `data/llm_system_prompts_lookup.csv` | Prompt and instruction-template lookup metadata. |
| `data/llm_system_instruction_tuning_samples.csv` | Synthetic instruction-tuning style samples aligned to interactions. |

## Artifact files

| File | Role |
|---|---|
| `artifacts/decision_artifact.json` | Top-level evaluation verdict, known issues, scope notes, and generated output references. |
| `artifacts/routing_backtest_summary.csv` | Held-out routing policy backtest summary. |
| `artifacts/routing_policy_use_case.csv` | Candidate use-case routing policy table. |
| `artifacts/drift_report.csv` | Early-vs-recent feature drift evidence. |
| `artifacts/triage_threshold_policy.json` | Selected triage threshold and policy metadata. |
| `artifacts/triage_threshold_curve.csv` | Threshold curve for review load, cost, precision, recall, and confusion-matrix tradeoffs. |
| `artifacts/triage_baseline_comparison.csv` | Comparison between baseline triage policies. |
| `artifacts/triage_actions_preview.csv` | Row-level review queue preview generated by the notebook workflow. |

## Validation performed by the app

At startup, `src.data.load_bundle()` validates:

- required files exist;
- CSV artifacts are non-empty;
- JSON artifacts are objects;
- minimum required columns are present;
- core numeric telemetry fields are non-negative;
- timestamps can be parsed;
- `interaction_id` values are unique;
- interaction `session_id` values exist in the session summary table;
- interaction `user_id` values exist in the user summary table;
- interaction `prompt_id` values exist in the prompt lookup table;
- instruction sample `interaction_id` values exist in the interaction telemetry table;
- instruction sample `session_id` values exist in the session summary table;
- instruction sample `user_id` values exist in the user summary table;
- instruction sample `prompt_id` values exist in the prompt lookup table;
- triage action preview rows reference known interaction IDs;
- bundled file SHA256 values and CSV dimensions match `docs/artifact_manifest.md`.

## Regeneration policy

When regenerating artifacts from the companion notebook:

1. Re-run the notebook end-to-end.
2. Copy only the expected files into `data/` and `artifacts/`.
3. Run local checks:

```bash
python scripts/run_tests.py
python -m ruff check app.py src tests
```

4. Update `docs/artifact_manifest.md` with the new row counts and SHA256 values.
5. Update `CHANGELOG.md` with the artifact change.
6. Re-check the dashboard manually with the default sidebar filters.

## Important interpretation rule

Live sidebar filters support investigation and candidate review. Held-out artifact verdicts remain the audit trail, which keeps evidence review separate from rollout approval.
